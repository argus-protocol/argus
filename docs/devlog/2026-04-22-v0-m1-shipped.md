# 2026-04-22 — v0 M1 shipped: AgentKit adapter, policy engine, Discord, dashboard

First dev log entry. Argus is now a working vertical slice — one adapter,
one policy, one alerter, one dashboard — end-to-end against a local anvil
node. Everything below is already in `main`.

## What shipped

- **AgentKit adapter** (`src/adapters/agentkit.ts`). An `ArgusWalletProvider`
  subclasses Coinbase AgentKit's `EvmWalletProvider` and wraps the inner
  provider. Every `sendTransaction` / `nativeTransfer` / `signTypedData`
  call emits a normalised event into the policy pipeline before returning.
  The observation point is the wallet boundary, not the action boundary —
  so it catches any code path an agent can use to broadcast a tx, including
  ones AgentKit doesn't know about yet.
- **Policy engine + daily spend cap** (`src/policies/spend-cap.ts`). Pure
  functions: `evaluate(event, state) → {state, decision}`. The cap policy
  sums native-value wei per UTC day and alerts on breach. Event schema is
  a `zod` discriminated union, validated at every ingress.
- **SQLite event + alert + policy-state store** (`src/storage/sqlite.ts`).
  WAL mode. `better-sqlite3` for v0. The `Store` interface is narrow on
  purpose — Postgres or DuckDB swap in later without touching callers.
- **Discord webhook alerter** (`src/alerts/discord.ts`). Severity-coloured
  embeds with policy, agent, chain, wallet, to, and value fields.
  Webhook URL is restricted to official `discord.com` hosts so a
  misconfigured env var can't leak alert payloads to a third party.
- **Dashboard** (`src/dashboard/*`). hono + SSR HTML + Tailwind CDN + SSE
  live refresh. Three pages: Fleet (agent list with 24h stats), Agent
  detail (events + alerts), Alerts feed with deep-linkable `/alerts/:id`
  that Discord embeds point at directly. Mobile card stack below the
  `sm:` breakpoint; primary mobile use case is alert triage.
- **End-to-end demo** (`scripts/demo-agentkit.ts`). Spawns a local anvil,
  runs four `native_transfer` actions through a real AgentKit surface,
  breaches the 1 ETH/day cap on the fourth tx, fires the Discord alert.
  Runnable as `npm run demo:agentkit`.

## Three design decisions that will outlive v0

**1. Wallet-level interception, not action-level.** AgentKit exposes two
natural observation seams: wrap each `Action.invoke` (call it α) or
subclass `EvmWalletProvider` (β). β is one level lower and bypass-proof
for the spend-cap policy — anything that broadcasts a tx goes through
`sendTransaction`, regardless of whether the agent used an AgentKit
action, raw viem, a LangChain tool, or handwritten code. α gives richer
semantic context ("agent called `swap` with these args") which we'll
want for allowlist and anomaly policies; it lands in v0.2 alongside β,
not replacing it.

**2. Observe-only for M1, synchronous gate in v0.3.** The adapter could
throw from `sendTransaction` to block a tx, but M1 just records and
alerts. Reason: the prevention surface has real contract design around
it — custom errors that LangChain and Vercel's AI SDK can surface
cleanly to the LLM, an `ARGUS_ENFORCE=false` escape hatch, timeouts on
policy evaluation. Building that on top of a noisy spend-cap policy
would calibrate the policy on the wrong signal. Observe first, cap-tune
on real traffic, then gate.

**3. SSR HTML dashboard, not SPA.** Considered React/Next, went with hono
+ plain templates + Tailwind CDN. The JSON API (`/api/agents`,
`/api/alerts`, `/api/stream`) is designed from day 1, so if an
integrator wants a bespoke SPA frontend later it's a frontend-only swap
— storage, policies, event pipeline don't change. The cost of writing
the SSR version first was ~1 day; the cost of writing the SPA version
first would have been the same day plus bundle setup, routing, and
state management, for a v0 read-only monitoring UI.

## What the code review caught

After the dashboard landed, a review pass flagged four real issues and
logged three SQL scalability risks for v0.2:

- `/agents/:id` returned 404 when deep-linked to a known agent with no
  recent rows — empty state is a valid view. Fixed.
- SSE client-side reload guard skipped the first real update after an
  empty-DB start because of a `last && ...` truthiness check on `0`.
  Fixed.
- Discord webhook URL validation accepted any `https://` host. Tightened
  to `discord.com` / `discordapp.com` / `ptb.` / `canary.` prefixes.
- Alert detail page for non-`tx_send` event kinds was too sparse.
  Enriched.

The SQL risks — full-table `json_extract` scans, missing composite
indexes, `ROW_NUMBER` over the full events table on every fleet render
— are documented in a private deferred-items log and queued for v0.2.
Not an issue at hundreds of events; matters at 10K+.
